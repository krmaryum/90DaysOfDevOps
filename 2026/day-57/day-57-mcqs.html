Day 57 MCQs – Resource Requests, Limits, and Probes

25 multiple-choice questions based on Day 57: resource requests, limits, OOMKilled behavior, Pending Pods, and Kubernetes probes.

Q1

What is the main purpose of resource requests in Kubernetes?

Show explanation

Requests define the minimum CPU and memory Kubernetes uses when scheduling a Pod onto a node.

Q2

What is the main purpose of resource limits in Kubernetes?

Show explanation

Limits are enforced at runtime and prevent a container from using more than the configured CPU or memory.

Q3

Which resource is compressible in Kubernetes?

Show explanation

CPU is compressible, so when a container exceeds its CPU limit it is usually throttled instead of killed.

Q4

Which resource is incompressible in Kubernetes?

Show explanation

Memory is incompressible, so exceeding the memory limit can result in the container being killed.

Q5

A Pod with requests lower than limits belongs to which QoS class?

Show explanation

If requests and limits are set but are not equal, Kubernetes assigns the Pod to the Burstable QoS class.

Q6

A Pod with no requests or limits set belongs to which QoS class?

Show explanation

Pods with no resource requests or limits are classified as BestEffort.

Q7

If requests are equal to limits for CPU and memory, the QoS class is:

Show explanation

A Pod is Guaranteed only when CPU and memory requests exactly match their limits.

Q8

What does 100m CPU mean?

Show explanation

CPU in Kubernetes uses millicores, so 100m means one-tenth of a CPU core.

Q9

What does an OOMKilled container usually indicate?

Show explanation

OOMKilled means the container used too much memory and was terminated.

Q10

What exit code is commonly associated with an OOMKilled container?

Show explanation

Exit code 137 usually means the process was killed with SIGKILL, which is common for OOMKilled containers.

Q11

Why can a Pod stay in Pending state forever?

Show explanation

If no node can satisfy the resource requests, the scheduler leaves the Pod in Pending.

Q12

Which command is most useful for checking why a Pod is Pending?

Show explanation

The Events section in kubectl describe pod usually explains scheduling failures.

Q13

What does a liveness probe do when it fails repeatedly?

Show explanation

Repeated liveness probe failures cause Kubernetes to restart the container.

Q14

What does a readiness probe control?

Show explanation

Readiness probes decide whether a Pod should be included in Service endpoints and receive traffic.

Q15

What happens when a readiness probe fails?

Show explanation

A readiness failure marks the Pod as NotReady and removes it from endpoints, but does not restart it.

Q16

What happens when a startup probe is running?

Show explanation

Startup probes give slow applications time to initialize by delaying liveness and readiness checks.

Q17

Why is a startup probe useful?

Show explanation

Startup probes protect slow-starting applications from premature restarts.

Q18

If a startup probe has periodSeconds: 5 and failureThreshold: 12, how much startup time is allowed?

Show explanation

The startup budget is periodSeconds multiplied by failureThreshold, which is 5 × 12 = 60 seconds.

Q19

If the startup probe failureThreshold were too low, what would most likely happen?

Show explanation

A low startup probe threshold can cause the container to be killed before it completes initialization.

When the Pod became NotReady, the Service endpoints became empty.
